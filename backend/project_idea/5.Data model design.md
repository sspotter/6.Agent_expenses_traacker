# 1. Core Design Philosophy

### Key principles

1. **Days never change once recorded**
    
2. **Money movements are transactions**
    
3. **Settlements are first-class records**
    
4. **Summaries are calculated, not stored**
    

This avoids bugs and accounting confusion later.

# 3. DailyCollection (Most Important Table)

Represents **what happened on that specific day**.

### DailyCollection

|Field|Type|Notes|
|---|---|---|
|id|UUID||
|user_id|UUID||
|date|DATE|Unique per user|
|expected_amount|DECIMAL|Default daily amount|
|collected_amount|DECIMAL|0 → expected|
|status|ENUM|FULL / PARTIAL / MISSED|
|notes|TEXT|Optional|
|created_at|TIMESTAMP||
|updated_at|TIMESTAMP||

### Rules

- `status` is derived but **stored** for speed & clarity
    
- Partial if `0 < collected < expected`
    
- Missed if `collected = 0`
    
- Full if `collected = expected`
    

✔ This table is **immutable across months** (no editing past months)

---

# 4. Outstanding Balance (Derived, Not Stored)

Outstanding is calculated as:

```
SUM(expected_amount - collected_amount)
WHERE status != FULL
MINUS settlements applied
```

No table needed — avoids sync bugs.

---

# 5. Settlement (Key to Your Problem)

Represents **money received later** to clear old partials.

### Settlement

|Field|Type|Notes|
|---|---|---|
|id|UUID||
|user_id|UUID||
|settlement_date|DATE|Date money was received|
|amount|DECIMAL||
|applied_to_month|YYYY-MM|Reference month|
|applied_to_day|DATE (nullable)|If specific|
|allocation_type|ENUM|OLDEST_FIRST / SPECIFIC|
|notes|TEXT|Optional|
|created_at|TIMESTAMP||

### Important Rules

- Settlement always belongs to **current month income**
    
- It _references_ old data, but does not modify it
    
- Can be split later if needed

# 10. Data Flow Example (Real Scenario)

### March 5

```
expected = 100
collected = 50
status = PARTIAL
```

### April 2 (remaining paid)

```
Settlement:
amount = 50
settlement_date = 2026-04-02
applied_to = March 5
```

### Result

- March 5 stays partial
    
- March outstanding reduces
    
- April income increases
    
- Audit intact ✔

# 11. Why This Model Works Long-Term

✔ Supports unlimited months  
✔ Handles late payments cleanly  
✔ No historical rewriting  
✔ Easy to audit  
✔ Easy to visualize  
✔ Easy to extend (multi-user, reports, exports)

---

# 12. What NOT to do (Important)

❌ Don’t store rolling balances  
❌ Don’t auto-edit old days  
❌ Don’t mix expenses into daily records  
❌ Don’t collapse settlement into daily collection
