# Core Insight (Key Design Shift)

A **payment being completed later** is **not an edit of the past** —  
it is a **new transaction that settles an old balance**.

So instead of “editing” a previous month, we **record a settlement**.

This is a very common accounting pattern and works perfectly for your case.

---

# Revised UX Principle (Very Important)

❌ Do NOT edit past daily cells  
✅ Allow **settlements against previous outstanding balances**

This keeps history intact **and** lets you finish partial payments later.

---

# How This Works Conceptually

### Before

- March 5 → Partial: 50 / 100
    
- Outstanding from March: 50
    

### Later (in April)

- You receive the remaining 50
    

Instead of changing March 5:

- You record **a settlement payment in April**
    
- That settlement is **linked to March**

# UX Design for Settling Past Partials

## 1. Monthly Summary Enhancement

Add a new line:

```
Outstanding: 450
Settled later: 120
Remaining: 330
```

This immediately explains what happened.

## 2. Settlement Entry Flow (UX)

### Where user initiates it

From **current month only**:

- Click on:
    
    - Outstanding amount
        
    - Or a “Settle Past Balance” button

## 3. How Past Month Displays Settlements (Read-Only but Informative)

In March (read-only):

### Day Cell Visual Update

```
🟡 50 / 100
✔ Settled in April
```

Or with subtle badge:

```
🟡 50
↳ Settled later
```

The original data is untouched.

---

## 4. Rules for Settlement UX

### Rules

- Settlement **cannot exceed outstanding**
    
- Settlement is always dated **today**
    
- Settlement always belongs to **current month income**
    
- Settlement references old month/day
    

---

## 5. What Happens in Reports (UX Clarity)

### Daily view

- Original partial stays partial
    

### Monthly summary

- Outstanding decreases
    
- “Settled later” increases
    

### Current month income

- Includes settlement amount
    

This avoids confusion like:

> “Why does March income change?”

---

# Revised Rule for “Read-Only Past Months”

### Updated UX Rule

> Past months are **non-editable**, but **settlements can be applied from future months**

This preserves:  
✔ Audit trail  
✔ Trust  
✔ Real-world payment behavior

---

# Alternative (Less Recommended)

Allow “unlock month” with admin password.

❌ Risky  
❌ Easy to accidentally rewrite history  
❌ Hard to audit

Your settlement approach is far superior.
