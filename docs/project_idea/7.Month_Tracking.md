# 📄 NEXT PHASE: Debt Visibility, Expense Payments & Settlement Refinement

## 1. Context & Current Problems

We have implemented:

* Daily income tracking
* Partial payments
* Settlements for late income
* Weekly expenses
* Read-only past months

However, **real-world usage revealed critical gaps**.

---

## 2. Identified Problems (Very Important)

### 2.1 Past Debt Is Invisible in Future Months ❌

**Current behavior**

* When viewing February
* Outstanding balance exists
* But user cannot see:

  * Which days caused the debt
  * Which expenses contributed
  * Which partial payments are still open

**Why this is a problem**

* Users lose trust
* Debt feels “magical”
* No auditability
* Impossible to decide what to pay first

👉 **Debt without breakdown is unusable**

---

### 2.2 Expenses Can Only Be “Settled” ❌

**Current behavior**

* Only one mechanism exists: Settlement
* Settlement is designed for **income shortfall**
* Expenses are treated as passive deductions

**Real-world need**

* Expenses may be:

  * Paid immediately
  * Paid later
  * Paid in parts
* Paying an expense is NOT the same as settling income

👉 **Expenses need their own payment lifecycle**

---

### 2.3 Settlement Is Overloaded ❌

Settlement is currently doing too much:

* Clearing income partials
* Acting like an expense payment
* Acting like a debt payment

This causes:

* Incorrect calculations
* UI confusion
* Double-counting risk

👉 **Settlement must be narrowed to ONE purpose**

---

## 3. Core Design Correction (Key Decision)

### 🔑 Separation of Concerns (Non-Negotiable)

| Concept         | What it represents                       |
| --------------- | ---------------------------------------- |
| DailyCollection | Income expected vs collected on that day |
| Settlement      | Late income received later               |
| Expense         | Cost incurred                            |
| ExpensePayment  | Money paid to clear an expense           |
| Debt View       | A READ-ONLY breakdown, not a table       |

If a concept moves money → it is a **transaction**
If it explains money → it is a **view**

---

## 4. New Required Capability: Debt Explorer (READ-ONLY)

### 4.1 Debt Must Be Visible Across Months

When viewing **any month**, the user must be able to:

* See **total outstanding**
* Expand to see:

  * Partial income days
  * Missed income days
  * Unpaid expenses
* See:

  * Original date
  * Original amount
  * Amount paid so far
  * Remaining balance

This is NOT editing — this is **financial transparency**.

---

### 4.2 Debt Explorer UI Concept

```
Outstanding Balance: 1,020

▼ Income Debt
  • Feb 15 – Partial – 40 remaining
  • Feb 21 – Missed – 100 remaining

▼ Expense Debt
  • Fuel (Week 2) – 80 unpaid
  • Maintenance – 200 unpaid
```

✔ Read-only
✔ Clickable for details
✔ Filterable by type / month

---

## 5. New Concept: Expense Payments (NOT Settlements)

### 5.1 Why Expense Payments Must Exist

Expenses are obligations, just like income shortfalls.

But:

* They are NOT income
* They should NOT affect “collected”
* They should NOT use settlements

👉 Expenses require their own payment tracking.

---

### 5.2 Expense Lifecycle

| Stage                  | Meaning       |
| ---------------------- | ------------- |
| Expense created        | Cost incurred |
| Expense unpaid         | Debt exists   |
| Expense partially paid | Debt reduced  |
| Expense fully paid     | Closed        |

---

### 5.3 New Entity: ExpensePayment (Conceptual)

Expense payments:

* Are dated
* Reduce expense debt
* Do NOT count as income
* Affect net cash flow

This mirrors Settlement, but for expenses.

---

## 6. Settlement Refined (Important Narrowing)

### Settlement is ONLY for:

✔ Income that was expected earlier
✔ Collected later
✔ Applied to DailyCollection gaps

### Settlement is NOT for:

❌ Paying expenses
❌ Fixing net balance manually
❌ Editing past days

This simplification will fix many bugs.

---

## 7. Correct Mental Model (For UI & Calculations)

```
Money IN
 ├── DailyCollection (on the day)
 └── Settlement (late income)

Money OUT
 ├── Expense (incurred)
 └── ExpensePayment (paid later)

Debt
 ├── Income debt (partials & missed)
 └── Expense debt (unpaid expenses)
```

No overlap. No shortcuts.

---

## 8. Calculation Rules (High Level)

### Outstanding Income Debt

```
SUM(expected - collected)
- SUM(settlements applied)
```

---

### Outstanding Expense Debt

```
SUM(expense.amount)
- SUM(expense_payments)
```

---

### Net Position (Any Period)

```
Income (Daily + Settlement)
- Expenses paid
```

⚠️ Unpaid expenses affect **debt**, not income.

## 9. UX Changes Required Next

### 9.1 New Panels / Views

* Debt Explorer (global, cross-month)
* Expense detail with payment history
* Clear distinction:

  * “Settle income”
  * “Pay expense”

---

### 9.2 New Actions

| Action             | Applies to           |
| ------------------ | -------------------- |
| Settle past income | DailyCollection debt |
| Pay expense        | Expense debt         |
| View debt history  | Read-only            |

---

## 10. What We Are NOT Doing (Yet)

❌ No automatic prioritization
❌ No interest / penalties
❌ No ledger exports
❌ No accounting journal UI

Keep system understandable first.

---

## 11. Definition of Done for This Phase

This phase is complete when:

✔ Past debt is visible in any month
✔ Income debt and expense debt are clearly separated
✔ Settlements only affect income
✔ Expenses can be paid independently
✔ No double-counting occurs
✔ UI always explains “why” a number exists

---

## 12. Next Concrete Implementation Tasks

1. Design **Debt Explorer UI**
2. Extend data model for **ExpensePayment**
3. Update calculations to separate:

   * income debt
   * expense debt
4. Update monthly summary to show:

   * income outstanding
   * expense outstanding
5. Update prompts / AI instructions accordingly

## Final Note (Important)

If users ever ask:

> “Where did this number come from?”

The system must always be able to **show the answer**.

That is the standard we are designing toward.
