# Worker Rate & Expected Amount – Implementation Specification

## Problem Statement
Currently, the system assumes a fixed **expected daily amount = 100** for all workers. This is not realistic.

Requirements:
- Each worker can have a **different rate**
- A worker’s rate **can change over time**
- A rate **can change between months or during a month**
- **Past days must keep their original rate**
- Reports, settlements, and summaries must remain historically correct

This document defines the **correct, future-proof model** for worker rates and expected amounts.

---

## Core Design Principle (Non‑Negotiable)

> **Rates are historical data, not mutable configuration.**

Once a day is calculated with a rate, that rate must never change retroactively unless explicitly edited by an admin.

---

## Data Model

### 1. worker_rates table
Stores the rate history for each worker.

```sql
CREATE TABLE worker_rates (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  worker_id UUID NOT NULL REFERENCES workers(id) ON DELETE CASCADE,
  rate_amount NUMERIC(10,2) NOT NULL,
  effective_from DATE NOT NULL,
  effective_to DATE,
  created_at TIMESTAMPTZ DEFAULT now()
);
```

Rules:
- `effective_to IS NULL` means **current active rate**
- Rates must **never overlap** for the same worker
- A new rate automatically closes the previous one

---

### 2. daily_collections (update)
Each day **stores the rate used at that time**.

```sql
ALTER TABLE daily_collections
ADD COLUMN expected_amount NUMERIC(10,2) NOT NULL;
```

This value is copied from `worker_rates.rate_amount` **at creation time**.

---

## Rate Resolution Logic

When creating a daily record:

1. Find active rate:
```sql
SELECT rate_amount
FROM worker_rates
WHERE worker_id = :workerId
  AND effective_from <= :date
  AND (effective_to IS NULL OR effective_to >= :date)
LIMIT 1;
```

2. Store returned value into `daily_collections.expected_amount`

---

## Changing a Rate

### Admin Action: Change Worker Rate

Steps:
1. Close current rate:
```sql
UPDATE worker_rates
SET effective_to = CURRENT_DATE - 1
WHERE worker_id = :workerId
  AND effective_to IS NULL;
```

2. Insert new rate:
```sql
INSERT INTO worker_rates (worker_id, rate_amount, effective_from)
VALUES (:workerId, :newRate, CURRENT_DATE);
```

---

## Mid‑Month Changes (Critical Behavior)

If a rate changes mid‑month:
- Days **before** the change keep the old rate
- Days **after** the change use the new rate
- Monthly totals automatically reflect the mix

No recalculation required.

---

## Editing Past Rates (Admin Only)

Admin may:
- Edit `expected_amount` on a specific day
- Insert a corrective historical rate (advanced)

⚠️ These actions must be **explicit and deliberate**.

---

## UI / UX Requirements

### Worker View
- Show current rate
- Show "Rate effective since <date>"

### Admin View
- Rate history table per worker
- Button: "Change Rate"
- Warning if rate is changed mid‑month

---

## Reporting & Calculations

All calculations must use:
- `daily_collections.expected_amount`

Never calculate expected income from a global or current rate.

---

## Definition of Done

- [x] Rates stored historically
- [x] Daily records store expected amount
- [x] Mid‑month rate changes supported
- [x] Past data remains unchanged
- [x] Admin can change rates safely
- [x] Reports remain accurate

---

## Why This Design Works

- Eliminates retroactive bugs
- Supports real‑world pay changes
- Makes reports legally & financially trustworthy
- Scales to bonuses, penalties, or contracts later

---

## Explicitly Out of Scope (For Now)

- Automatic rate inflation
- Performance‑based rates
- Per‑day custom overrides UI

These can be layered later without breaking this model.
