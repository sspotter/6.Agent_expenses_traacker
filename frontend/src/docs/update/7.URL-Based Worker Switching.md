

# Implementation Spec: URL-Based Worker Switching

## Objective

Enable **worker switching via URL**, where:

* Each worker has **fully isolated data**
* The active worker is derived from the **URL**
* Password protection is applied **only when required**
* A **default worker** is always available

Example URLs:

* `/` → default worker
* `/user1`
* `/ahmed`
* `/worker-uuid` (optional future-proofing)

---

## Core Mental Model (Non-Negotiable)

> **The URL defines the active worker context.**

* No URL = default worker
* Changing the URL = switching worker
* Worker switching is a **full data re-hydration**
* Password only gates *access*, not identity

---

## Routing Rules

### Valid Routes

```text
/
 /:workerSlug
```

### Behavior

| URL        | Result                        |
| ---------- | ----------------------------- |
| `/`        | Load default worker           |
| `/user1`   | Load worker with slug `user1` |
| `/unknown` | Redirect to `/`               |

---

## Worker Identity Resolution

### Worker Fields (Required)

```ts
Worker {
  id: uuid
  name: string
  slug: string   // unique, URL-safe
  password_hash?: string | null
}
```

### Slug Rules

* Lowercase
* URL-safe
* Unique
* Generated from name (with fallback suffix)

Example:

```text
"John Smith" → "john-smith"
```

---

## Worker Resolution Flow (Critical)

When route changes:

1. Extract `workerSlug` from URL
2. Query workers table:

   ```sql
   select * from workers where slug = :workerSlug and is_active = true
   ```
3. If not found:

   * Redirect to `/`
   * Load default worker
4. If found:

   * Continue to access check

---

## Default Worker Rules

* Exactly **one default worker**
* No password
* Always accessible

### Database Rule

```sql
is_default boolean not null default false
```

### Guarantee

* On app load with `/`
* On invalid URL
* On access denied

➡️ Always fall back to default worker

---

## Password Gating Logic (UI-Only)

### Rules

* Password is optional
* Stored hashed
* Checked **before activating worker**
* DB does not enforce auth

### Access Flow

```text
URL change →
Resolve worker →
If password_hash is NULL → allow
If password_hash exists → show password modal
```

### Password Modal

* Input field
* Cancel → redirect to `/`
* Success → unlock worker for session

### Session Behavior

* Unlock stored **in memory only**
* Refresh = locked again
* URL alone is not enough to bypass password

---

## Worker Activation (Single Source of Truth)

Only after access is granted:

```ts
setActiveWorker({
  id,
  name,
  slug,
  avatarUrl
})
```

This triggers:

* Data refetch
* Skeleton UI
* Derived state recalculation

---

## Data Fetching Contract (MANDATORY)

Every data hook must accept `workerId`.

Example:

```ts
useDailyCollections(workerId)
useWeeklyExpenses(workerId)
useSettlements(workerId)
```

❌ No hook may read without worker scope
❌ No shared cache across workers

---

## Worker Switching UX Flow

### Switching Methods

1. URL navigation
2. “Switch User” modal
3. Browser back/forward

All must behave identically.

### On Switch

1. Clear previous worker data
2. Show skeleton UI
3. Resolve worker
4. Validate password (if needed)
5. Fetch worker-scoped data
6. Render UI

---

## Skeleton Loading Rules

Skeletons must cover:

* Calendar grid
* Weekly cards
* Monthly summary
* Debt explorer

❌ No spinners
✅ Skeletons only

---

## Error & Edge Case Handling

| Case                 | Behavior             |
| -------------------- | -------------------- |
| Invalid slug         | Redirect `/`         |
| Wrong password       | Stay locked          |
| Worker deleted       | Redirect `/`         |
| Switching mid-action | Cancel action        |
| Direct URL access    | Apply password rules |

---

## Security Model (Explicit)

* No Supabase Auth
* No sessions
* No cookies
* No RLS required (optional later)
* Password = local access gate only

---

## Final Guarantee (Must Be True)

* Each worker sees **only their own data**
* URL always reflects active worker
* Default worker is always accessible
* Switching feels instant and safe
* No historical data corruption possible

---

## Optional Future Extensions (Do NOT implement now)

* `/w/:workerId` routing
* Shareable read-only links
* Per-worker RLS
* Avatar uploads

---

### TL;DR for the AI Agent

> “Treat workers like **URL-scoped data universes**.
> The URL selects the worker, the password unlocks it, and every query is scoped.”
