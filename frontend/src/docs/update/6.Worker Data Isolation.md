Ah — this is a **classic and very important multi-tenant bug**, and you already described the symptoms perfectly 👌
Good news: nothing is “broken”, but your system is currently **worker-agnostic**, not worker-scoped.

Let’s turn this into a **clear implementation task list** and explain *why* this happened and *how to fix it permanently*.

---

## What Went Wrong (Root Cause)

You did **only half** of multi-worker isolation 👇

### 1️⃣ Data is NOT scoped at write time

* When you add a payment / expense, **`worker_id` is not being set**
* So rows are created with:

  ```sql
  worker_id = NULL
  ```
* When you switch users, both users see the same records (NULL = global)

### 2️⃣ Queries are NOT scoped at read time

* Your UI queries are probably like:

  ```sql
  select * from weekly_expenses;
  ```
* Instead of:

  ```sql
  select * from weekly_expenses where worker_id = activeWorkerId;
  ```

### 3️⃣ Existing data was never migrated

* Old records have no `worker_id`
* So all workers “share” legacy data

This is **expected behavior** with the current schema.

This is a **full implementation phase**, touching:

* DB constraints

# Implementation Phase: Worker Data Isolation (CRITICAL)

## Phase 1: Decide the Isolation Rule (Non-Negotiable)

> **Every financial record MUST belong to exactly one worker**

That means:

* ❌ No NULL `worker_id`
* ❌ No global data
* ❌ No default fallback at query time

---

## Phase 2: Enforce at Database Level (Fix Writes)

### 2.1 Make `worker_id` REQUIRED

⚠️ Do **not** do this yet if you have existing data — migration comes first.

```sql
alter table daily_collections
  alter column worker_id set not null;

alter table weekly_expenses
  alter column worker_id set not null;

alter table settlements
  alter column worker_id set not null;

alter table settlement_allocations
  alter column worker_id set not null;

alter table expense_payments
  alter column worker_id set not null;
```

This ensures:

* If the app forgets `worker_id` → **DB rejects the insert**
* Bugs become visible immediately

---

## Phase 3: Migrate Existing Data (One-Time)

Pick a **default worker** (usually the first one).

```sql
-- Find default worker
select id from workers order by created_at asc limit 1;
```

Then:

```sql
update daily_collections
set worker_id = '<DEFAULT_WORKER_ID>'
where worker_id is null;

update weekly_expenses
set worker_id = '<DEFAULT_WORKER_ID>'
where worker_id is null;

update settlements
set worker_id = '<DEFAULT_WORKER_ID>'
where worker_id is null;

update settlement_allocations
set worker_id = '<DEFAULT_WORKER_ID>'
where worker_id is null;

update expense_payments
set worker_id = '<DEFAULT_WORKER_ID>'
where worker_id is null;
```

Only **after this**, enforce `NOT NULL`.

## Phase 5: Fix ALL Write Operations (This Is Why Payments Failed)

Every insert **must include `worker_id`**.

### Example: Daily Payment

❌ Wrong

```ts
insert into daily_collections (date, collected)
```

✅ Correct

```ts
insert into daily_collections (
  worker_id,
  date,
  collected
) values (
  activeWorkerId,
  date,
  amount
)
```

This applies to:

* Daily payments
* Weekly expenses
* Settlements
* Expense payments
* Payment events

> **If worker_id is missing, the DB should now reject it**

That’s good.

---

## Phase 6: Fix ALL Read Queries (Why User2 Saw User1 Data)

Every query must be scoped.

### Example

❌ Wrong

```ts
select * from weekly_expenses;
```

✅ Correct

```ts
select * from weekly_expenses
where worker_id = activeWorkerId;
```

This applies to:

* Grid data
* Monthly summary
* Debt explorer
* Weekly cards
* Reports

---

## Phase 7: Switching Workers = Full Data Re-Hydration

When user switches worker:

1. Update `ActiveWorkerContext`
2. Clear derived state
3. Show skeleton UI
4. Re-fetch ALL data with new `worker_id`
5. Recompute derived metrics

⚠️ Never reuse cached data across workers.

---

## Phase 8: Password Logic (Still UI-Only)

* Password only gates **switching**

* No RLS yet (optional later)

## Phase 9: Optional but STRONGLY Recommended (Safety Net)

### Add Composite Indexes

```sql
create index idx_daily_collections_worker_date
on daily_collections (worker_id, date);

create index idx_weekly_expenses_worker_week
on weekly_expenses (worker_id, week_start_date);
```

This keeps switching instant.

## Final Mental Model (THIS is the key)

Right now your system is:

> ❌ **Multi-worker UI, single-worker data**

What you want is:

> ✅ **Single active worker, fully isolated data universe**

Each worker is basically:
