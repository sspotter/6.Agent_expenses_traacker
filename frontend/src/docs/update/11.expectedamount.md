Perfect, this screenshot makes the requirement **very clear** 👍
What you’re describing is a **rate override hierarchy**, not just “editing expected”.

Below is a **clean, professional bullet-point implementation task list** that you can later turn into an MD file or tickets.

---

## 🎯 Goal

Allow **flexible expected amount changes** that:

* Can start from **any date** (mid-month supported)
* Do **not rewrite history**
* Allow **one-day exceptions**
* Keep reports, settlements, and admin logic correct

---

## 1️⃣ Core Concept (Very Important)

There are **3 levels of expected amount**, in this order of priority:

1. **Daily Override** (highest priority – single day only)
2. **Rate Period** (effective from a date onward)
3. **Fallback / Legacy Default** (only for migration, then removed)

> The UI may look simple, but the data model must respect this hierarchy.

---

## 2️⃣ Data Model Adjustments (Implementation Tasks)

* ✅ **Daily collections must store `expected_amount` explicitly**

  * This value is **snapshotted at creation**
  * Never recalculated automatically later

* ✅ **Worker rate changes are stored separately**

  * Rate has:

    * `worker_id`
    * `amount`
    * `effective_from_date`
    * `created_by_admin`
    * `created_at`

* ❌ Never update existing daily records when a rate changes

* ❌ Never infer expected amount dynamically from “current rate”

---

## 3️⃣ Changing Worker Rate (Admin Flow)

### UI Behavior

* Admin clicks **“Change Rate”**
* Modal requires:

  * New amount
  * **Effective start date**
* Clear helper text:

  > “This rate will apply to all future days starting from the selected date.”

### Logic

* Insert a new `worker_rate` record
* All **future daily records**:

  * Use new rate
* All **past daily records**:

  * Keep original expected amount

---

## 4️⃣ Editing Expected Amount for a Single Day (Day Override)

### UI Behavior (your screenshot flow)

* Admin clicks on **Expected Amount** in day modal
* Editable numeric input appears
* Helper text:

  > “This change applies only to this day.”

### Logic

* Update **only that daily record’s `expected_amount`**
* Mark record as:

  * `is_expected_override = true`
* Does NOT:

  * Affect worker rate
  * Affect other days
  * Affect future days

---

## 5️⃣ Mid-Month Rate Change Rules (Explicit)

* ✔️ Rate can change **any day**
* ✔️ Days before the effective date keep old rate
* ✔️ Days after use the new rate
* ✔️ Partial / missed / settled days remain untouched
* ✔️ Reports must sum stored `expected_amount`, not recompute

---

## 6️⃣ Reporting & Calculations (Critical)

* Monthly Expected =
  `SUM(daily.expected_amount)`
* Outstanding =
  `expected_amount - collected - settlements`
* Rate history is **never used directly** in reports
* Reports must work even if:

  * Rate changed 5 times in a month
  * Some days have overrides

---

## 7️⃣ UX Safeguards (Highly Recommended)

* Show a small badge/icon on days with overridden expected amount
* Show tooltip:

  > “Expected amount manually adjusted”
* In Admin:

  * Show rate history timeline per worker
* Prevent accidental bulk edits:

  * No “apply to past days” option

---

## 8️⃣ Migration Tasks

* Remove hardcoded `100`
* Backfill existing days with:

  * `expected_amount = 100`
* Create initial worker rate records for historical data
* Lock legacy behavior after migration

---

## 9️⃣ Definition of Done ✅

* A worker can:

  * Have different rates per month
  * Change rate mid-month
* Admin can:

  * Adjust a single day without affecting others
* Refreshing, switching workers, or reopening months:

  * Shows **correct expected amounts**
* No recalculation bugs
* No silent history mutation
