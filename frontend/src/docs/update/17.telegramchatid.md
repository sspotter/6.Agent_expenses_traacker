# Worker Telegram Notifications – Implementation Plan

## 1. Goal

Enable sending **monthly summaries** (and later other notifications) to **Telegram chats** associated with a worker.

Each worker:

* Can have **zero, one, or many Telegram chat IDs**
* Can **add** chat IDs
* Can **delete** chat IDs
* Uses these chat IDs when sending monthly summaries

No authentication with Telegram is required — we only store chat IDs.

---

## 2. Data Model Design

### 2.1 New Table: `worker_telegram_chats`

We **do NOT** store this directly on `workers` because:

* A worker can have multiple chat IDs
* We want clean add/remove history later if needed

```sql
create table worker_telegram_chats (
  id uuid primary key default gen_random_uuid(),

  worker_id uuid not null
    references workers(id)
    on delete cascade,

  chat_id text not null,

  label text, -- optional (e.g. "Personal", "Supervisor Group")

  is_active boolean not null default true,

  created_at timestamptz not null default now()
);
```

### Constraints

```sql
-- Prevent duplicate chat IDs per worker
create unique index uniq_worker_chat
on worker_telegram_chats(worker_id, chat_id);
```

## 4. Frontend Logic

### 4.1 Fetching Chat IDs

On worker modal open:

```ts
select *
from worker_telegram_chats
where worker_id = :workerId
order by created_at asc
```

Store in modal-local state.

---

### 4.2 Add Chat ID

Flow:

1. User submits chat ID
2. Insert into `worker_telegram_chats`
3. On success:

   * Append to local state
4. On error:

   * Show inline error (duplicate / invalid)

```ts
insert({
  worker_id,
  chat_id,
  label
})
```

---

### 4.3 Delete Chat ID

Flow:

1. User clicks delete
2. Soft-confirm (optional)
3. Delete row

```ts
delete
from worker_telegram_chats
where id = :id
```

Because of `on delete cascade`:

* Removing worker automatically removes all chat IDs

---

## 5. Monthly Summary Integration

### 5.1 Summary Target Resolution

When sending a monthly summary:

1. Load worker
2. Load active Telegram chats:

```sql
select chat_id
from worker_telegram_chats
where worker_id = :workerId
and is_active = true;
```

3. Send the same summary payload to **each chat ID**

---

### 5.2 Failure Rules

* One chat failure **does not block others**
* Log failures per chat ID
* Summary is considered sent if **at least one succeeds**

---

## 6. Telegram Service Layer

Create a dedicated helper:

```
lib/telegram/sendMessage.ts
```

Responsibilities:

* Accept `chat_id`
* Accept formatted message
* Call Telegram Bot API
* Return success/failure

This keeps:

* UI clean
* Monthly job logic clean
* Easy future reuse (daily, weekly, alerts)

---

## 7. Permissions & Security

* Only admins / authorized roles can:

  * Add chat IDs
  * Remove chat IDs
* Chat IDs are treated as **non-secret identifiers**
* No bot token ever exposed to frontend

---

## 8. Future-Proofing (Explicitly Supported Later)

This design supports later:

* Chat ID verification flow
* Per-chat notification preferences
* Daily / weekly summaries
* Group vs personal chats
* Audit trail for sent notifications

* ✅ One-to-many relationship handled properly
* ✅ Clean separation of concerns
* ✅ No schema hacks on `workers`
* ✅ Safe deletion
* ✅ Scales naturally to automation
