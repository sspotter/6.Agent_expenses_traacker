# Daily Grid Resolution – Implementation Specification

## 1. Purpose

This document defines the **single source of truth** for how each day is calculated, classified, and rendered inside the **DailyGrid**.

The goal is to ensure that:
- Settlements are visually and numerically reflected
- Surplus never auto-resolves days
- Historical daily records remain immutable
- UI state always matches financial reality

---

## 2. Core Principle

> **DailyGrid must never render directly from `daily_collections` alone.**

Instead, it must render from a **derived, resolved day model** that merges:
- Daily collection data
- Settlement allocations

---

## 3. Source Tables

### 3.1 `daily_collections`
Represents what happened **on the day itself**.

Used for:
- Expected amount
- Collected amount
- Attendance / work status

Must NEVER be mutated by settlements.

### 3.2 `settlements`
Represents money applied **after the fact**.

Used for:
- Resolving missed or partial days
- Explicit admin actions only

---

## 4. Resolved Day Model

For each worker and date, the system MUST derive the following model:

```
ResolvedDay {
  date
  base_expected_amount
  collected_amount
  settled_amount
  resolved_amount
  remaining_amount
  day_state
  is_expected_override
}
```

### 4.1 Field Definitions

- **base_expected_amount**
  - From `daily_collections.expected_amount`

- **collected_amount**
  - From `daily_collections.collected_amount`

- **settled_amount**
  - SUM of `settlements.amount`
  - Filtered by same `worker_id + date`

- **resolved_amount**
  ```
  collected_amount + settled_amount
  ```

- **remaining_amount**
  ```
  base_expected_amount - resolved_amount
  ```

---

## 5. Day State Classification

Each day MUST map to **exactly one** state.

### 5.1 Day States

| State | Meaning |
|-----|-------|
| `PAID_ON_TIME` | Collected >= expected on same day |
| `SETTLED_LATER` | Collected < expected but settlements completed it |
| `PARTIAL` | Collected > 0 but still unresolved |
| `PARTIAL_ZERO` | Expected > 0, collected = 0 |
| `MISSED_NO_WORK` | Expected = 0 (no work day) |
| `UNRESOLVED` | Expected > resolved |

### 5.2 Classification Rules (Priority Order)

1. If `base_expected_amount = 0` → `MISSED_NO_WORK`
2. If `collected_amount >= base_expected_amount` → `PAID_ON_TIME`
3. If `resolved_amount >= base_expected_amount` → `SETTLED_LATER`
4. If `collected_amount > 0` → `PARTIAL`
5. If `collected_amount = 0` → `PARTIAL_ZERO`
6. Else → `UNRESOLVED`

---

## 6. Visual Representation (CSS Rules)

DailyGrid cards MUST visually reflect `day_state`.

| Day State | Visual Treatment |
|--------|----------------|
| `PAID_ON_TIME` | 🟢 Green |
| `SETTLED_LATER` | 🟣 Purple + `resolved / expected` |
| `PARTIAL` | 🟡 Yellow |
| `PARTIAL_ZERO` | 🔴 Red |
| `MISSED_NO_WORK` | ⚪ Greyed |

---

## 7. Manual Override Indicator

### Rule
If `is_expected_override = true`:
- Display an **`M` badge** on the day card
- Badge persists regardless of payment or settlement status

### Purpose
- Allows admins to audit manual expectation changes
- Prevents silent data manipulation

---

## 8. Settlement Rules (Non-Negotiable)

- Settlements NEVER update `daily_collections`
- Settlements NEVER auto-apply surplus
- Settlements must target explicit dates
- `NO_WORK` days CANNOT be settled

---

## 9. Surplus Handling

- Over-collection creates **surplus only**
- Surplus is stored separately
- Surplus does NOT affect DailyGrid unless explicitly settled

---

## 10. Completion & Totals Calculations

### 10.1 Day Completion

```
resolved_amount / base_expected_amount
```

### 10.2 Monthly Completion

- Uses resolved values only
- Surplus excluded

---

## 11. Validation Scenarios

The implementation MUST correctly handle:

- Overpaid day creating surplus
- Missed day later settled
- No-work days excluded from settlements
- Mid-month rate change
- Manual expected override + settlement

---

## 12. Definition of Done

This spec is considered complete when:

- DailyGrid reflects settlements correctly
- Visual state matches financial truth
- No historical records are mutated
- Admin can visually audit all anomalies

---
