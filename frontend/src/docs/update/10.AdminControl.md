# Admin Panel – Implementation Specification

This document defines the **Admin Page** behavior, data visibility rules, permissions, and UI structure.
It is written as an **implementation-ready reference** for frontend, backend, and database work.

---

## 1. Role Model

### 1.1 Admin Definition

* Admin is a **special worker record**
* Admin exists in the same `workers` table
* Admin is identified by:

  * `role = 'admin'` OR
  * `is_admin = true`

### 1.2 Admin Capabilities

Admin can:

* View **all workers** (active + suspended)
* Edit **any financial data**
* Adjust **worker rates**
* Suspend / reactivate workers
* Remove or reset worker passwords
* Export & import all data

Admin cannot:

* Be suspended accidentally

---

## 2. Worker Lifecycle States

### 2.1 Worker Statuses

* `active`
* `suspended`

### 2.2 Suspended Workers

* Do NOT appear in normal worker switching
* DO appear in:

  * Admin page
  * Reports → "Suspended Workers" section
* Data remains visible (read-only unless reactivated)

---

## 3. Admin Page Layout

### 3.1 Global Admin Header

Displayed at the top of the admin page.

**Metrics:**

* Total workers

  * Active count
  * Suspended count
* Global financial totals:

  * Expected
  * Collected
  * Outstanding

**Controls:**

* Add worker
* Export data
* Import data
* Global settings (future)

---

## 4. Workers List Structure

### 4.1 Sectioning

* **Active Workers** (expanded by default)
* **Suspended Workers** (collapsed by default)

---

### 4.2 Worker Collapsed Card (Summary View)

Each worker card displays:

* Worker name
* Status badge (Active / Suspended)
* Current rate badge (e.g. `$100 / day`)
* Monthly snapshot:

  * Expected
  * Collected
  * Outstanding
  * Completion %

Purpose: quick scanning and comparison.

---

## 5. Expanded Worker View (Admin Mode)

When a worker card is expanded:

### 5.1 Worker Control Panel

**Display:**

* Name
* Status
* Password state (set / removed)
* Current rate

**Actions:**

* Suspend / Reactivate worker
* Remove password
* Reset password
* Switch to worker view (admin mode banner shown)

---

## 6. Rate Management (Critical)

### 6.1 Rate Rules

* Worker rate **CAN change mid-month**
* Past days MUST keep their original rate
* Rate changes must NOT recalculate historical data

### 6.2 Rate History Model

Conceptually required:

* Each worker has **rate history records**

Each rate record:

* `worker_id`
* `rate_amount`
* `effective_from` (date)
* `created_by` (admin)

### 6.3 Admin UI for Rate Change

* Display current active rate
* "Change Rate" action opens modal:

  * New rate value
  * Effective date (default = today)
  * Optional note

---

## 7. Financial Overview (Editable by Admin)

### 7.1 Summary

* Monthly Expected (based on rate history)
* Collected
* Outstanding
* Net balance

### 7.2 Admin Actions

* Add manual adjustment (+ / –)
* Correct payment
* Correct expense
* Mark items as forgiven (future)

All admin edits must be explicit (no silent changes).

---

## 8. Payments & Expenses Snapshot

Displayed inside expanded worker view:

* Recent payments (last N)
* Recent expenses
* Outstanding items highlighted

Each item shows:

* Amount
* Date
* Type
* Admin edit / delete controls

---

## 9. Reports Behavior

### 9.1 Report Sections

* Active Workers
* Suspended Workers (separate section)

### 9.2 Totals Toggle

Admin can choose:

* Include suspended workers
* Exclude suspended workers

---

## 10. Data Export & Import (Backup)

### 10.1 Export

Admin can export:

* All workers
* All rate history
* All payments
* All expenses
* All settlements & allocations

Export format:

* JSON (primary)
* CSV (optional future)

Export should include:

* Schema version
* Export timestamp

---

### 10.2 Import

Admin can import full dataset:

* Restore from backup
* Overwrite existing data OR import as new

Import safeguards:

* Schema compatibility check
* Dry-run validation
* Explicit confirmation required

---

## 11. Admin Mode UX Rules

* Clear banner when viewing worker as admin

  * "Viewing Worker X (Admin Mode)"
* Easy return to Admin dashboard
* Dangerous actions require confirmation

---

## 12. Security & Integrity Rules

* All admin actions are logged
* Rate changes are immutable once applied
* Financial edits must preserve auditability

---

## 13. Future Extensions (Non-Blocking)

* Full audit log page
* Worker performance charts
* Per-worker export
* Multi-admin support

---

## 14. Definition of Done

Admin page is complete when:

* Admin can view all workers
* Data is correctly isolated per worker
* Rate changes work mid-month without affecting history
* Suspended workers are visible but isolated
* Full data export & import works reliably
