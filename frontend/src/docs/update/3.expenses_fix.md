

# Phase 6: Calculation Integrity & Input Validation (Stability Phase)

**Goal:**
Eliminate calculation errors, broken validations, and silent UI failures.

---

## Settlement Input & Validation Fixes

* Fix settlement amount validation bug

  * Remove incorrect rounding / step constraint (`nearest 100.99`)
  * Allow **any decimal amount** within currency precision (e.g. 2 decimals)
  * Validate only:

    * `amount > 0`
    * `amount <= outstanding income debt`

* Align settlement input validation with:

  * `expected_amount`
  * remaining outstanding after allocations

* Ensure settlement modal:

  * Accepts manual amounts
  * Shows **remaining outstanding live** while typing
  * Blocks submission only on real violations

---

## Settlement Allocation Accuracy

* Verify allocation logic:

  * Oldest-first allocation distributes correctly
  * Partial allocations allowed
  * No over-allocation to a single day

* Add guardrails:

  * Prevent settlement allocation exceeding daily remaining balance
  * Prevent duplicate allocations on re-submit

* Add allocation preview before submit:

  * Show which days will be affected
  * Show remaining balance after settlement

---

## Expense Edit & Recalculation Fixes

* Fix expense edit not reflecting in:

  * Weekly totals
  * Monthly summary
  * Debt explorer

* Ensure expense updates:

  * Trigger recalculation of weekly net
  * Update expense debt (if unpaid)
  * Do NOT affect income or settlements

* Add optimistic UI update with rollback on failure

---

## Derived Summary Re-validation

* Re-verify all derived values:

  * Collected
  * Net
  * Outstanding income debt
  * Outstanding expense debt

* Add internal assertion checks:

  * No negative outstanding
  * No double-counting of settlements
  * No expense affecting collected income

---

# Phase 7: UX Productivity & Financial Workflow Enhancements

**Goal:**
Make daily + weekly operations faster and more intuitive without breaking accounting rules.

---

## Quick Weekly Expense Actions

* Add **“Quick Add Weekly Expense”** button beside each week column

  * User can define:

    * Default amount
    * Category (optional)
    * Auto-apply to selected week

* Support:

  * One-click add using last-used amount
  * Editable after creation
  * Visible immediately in weekly totals

* Ensure quick-add:

  * Creates a standard `expense` record
  * Does NOT auto-mark as paid
  * Appears in debt explorer if unpaid

---

## Expense Payment Workflow (UX Polish)

* Add explicit **“Pay Expense”** action on expenses

  * Separate from settlement UI
  * Clear wording (no financial ambiguity)

* Support:

  * Partial payments
  * Full payments
  * Multiple payments over time

* Show expense payment history inline:

  * Paid amount
  * Remaining balance
  * Payment dates

---

## Daily Entry UX Improvements

* Fix FULL / MISSED auto-fill behavior:

  * FULL → auto-fill collected = expected
  * MISSED → auto-fill collected = 0
  * Never require manual input for these states

* Prevent invalid state transitions:

  * Cannot mark FULL if outstanding exists on that day
  * Warn when converting PARTIAL → FULL

---

## Debt Explorer UX Enhancements

* Add filters:

  * Income debt only
  * Expense debt only
  * Month range
  * Oldest first / largest first

* Add drill-down:

  * Click debt item → see origin (day/week)
  * Show linked settlements or payments

---

## Feedback & Trust Indicators

* Add inline explanations:

  * “Why is this outstanding?”
  * “Where did this number come from?”

* Add subtle indicators:

  * “Fully settled”
  * “Partially settled”
  * “Unpaid”

---

## Phase 6 & 7 Completion Criteria

Phase 6 is complete when:

* No invalid settlement input errors
* Calculations are stable and predictable
* Expense edits always recalculate totals

Phase 7 is complete when:

* Weekly expenses can be added in one click
* Expense payments are distinct from settlements
* Daily operations require minimal friction
* Users never wonder *why* a number changed

---

## Important Guiding Rule (Carry Forward)

> **If a number changes, the UI must explain why.**
> If it cannot, the feature is incomplete.
