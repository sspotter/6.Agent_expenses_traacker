# Fix Daily Grid Not Reflecting Settlements – Implementation Tasks

## Problem Summary

After applying a **Settlement**:
- ✅ Outstanding balance updates correctly
- ✅ Monthly collected totals update correctly
- ❌ Daily Grid still shows:
  - `0 / 100`
  - Yellow (PARTIAL)

This creates a **false UX state** where the system is financially correct, but visually wrong.

Root cause: **Daily Grid is rendering raw `daily_collections` instead of resolved data**.

---

## Root Cause Analysis

### What is happening now
- Daily Grid reads directly from:
  - `daily_collections.collected_amount`
  - `daily_collections.expected_amount`
- Settlements are stored separately in:
  - `settlements`
  - `settlement_allocations`
- Grid does **not** aggregate settlements per day

Result:
> A day that is fully resolved via settlement still appears PARTIAL or MISSED.

---

## Core Principle (Must Be Enforced)

> **The Daily Grid must never render raw data.**

It must render a **Resolved Day**, derived from:
- Daily Collection
- Settlement Allocations
- Expected overrides
- Non-working day rules

---

## Target Architecture

### Introduce a `ResolvedDay` layer

Daily Grid must consume:
```
ResolvedDay {
  date
  expected_amount
  collected_direct
  collected_via_settlement
  total_collected
  outstanding
  resolution_status
  visual_state
  badges[]
}
```

No component may compute status inline.

---

## Resolution Status Rules

| Condition | Visual | Notes |
|---------|--------|------|
| Paid on time | Green | collected_direct === expected |
| Settled later | Purple | settlement_allocations > 0 |
| Partial | Yellow | 0 < total_collected < expected |
| Missed | Red | total_collected === 0 AND expected > 0 |
| No work | Grey | expected === 0 |
| Manual override | Badge `M` | expected override used |

---

## Implementation Tasks

### Task 1 — Aggregate settlements per day

Create selector or SQL view that returns:
```
SUM(settlement_allocations.amount)
GROUP BY target_date
```

This must be queryable per worker + month.

---

### Task 2 — Build `resolveDay()` function

Inputs:
- daily_collection
- settlement_allocations_for_day
- base_expected_rate

Outputs:
- ResolvedDay

This function is the **single source of truth**.

---

### Task 3 — Refactor DailyGrid to use ResolvedDay

- ❌ Remove status logic from UI
- ❌ Remove reliance on `collected_amount` only
- ✅ Render:
  - `total_collected`
  - `resolution_status`
  - `visual_state`

Grid should not know what a settlement is.

---

### Task 4 — Visual corrections

Apply styles strictly based on `ResolvedDay.visual_state`:
- Green → paid on time
- Purple → resolved via settlement
- Yellow → partial
- Red → missed
- Grey → no work

Add `M` badge if:
```
resolved.expected_amount !== base_rate
```

---

### Task 5 — Validation checklist

After implementation:
- [ ] Settled days turn purple
- [ ] `0 / 100` never appears on fully settled days
- [ ] Outstanding = 0 implies non-red/non-yellow
- [ ] Surplus does not affect grid
- [ ] Grey days are never settleable

---

## Explicit Non-Goals

- ❌ Do not update `daily_collections` during settlement
- ❌ Do not auto-apply surplus
- ❌ Do not recompute history

---

## Definition of Done

The Daily Grid must visually match financial reality.

If totals say resolved, the grid must say resolved.

Any mismatch is considered a **blocking bug**.
