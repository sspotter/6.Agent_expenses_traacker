# Worker Data Isolation & Switching – Professional Implementation Plan

This document defines the **production-grade implementation plan** to complete multi-worker support with **correct data visibility**, **strict isolation**, and **predictable switching behavior**.

This phase focuses on the **core missing link**:

> Workers can be switched, but **their data is not being reloaded correctly**.

---

## 1. Problem Statement (Precisely Defined)

### Current Behavior

* Worker switching (URL + password) works
* Active worker persists across refresh
* Records are saved correctly with `worker_id`

### Broken Behavior

* Switching workers does **not change displayed data**
* After refresh, UI may show empty state
* UI does not react to `activeWorkerId` changes

### Root Cause

> **Data hooks are not reactive to `activeWorkerId`.**

The system currently has:

* ✅ Worker Context
* ❌ Worker-scoped data layer

---

## 2. Non-Negotiable Architecture Rule

> **`activeWorkerId` must be a dependency of ALL data fetching and calculations.**

If a hook does not depend on `activeWorkerId`, it is invalid.

---

## 3. Phase 1 – Establish the Worker Data Contract

### Goal

Define a strict contract between:

* Worker Context
* Data Hooks
* UI Components

### Contract

Every data hook MUST:

```ts
useSomething(workerId: string | null)
```

Rules:

* If `workerId === null` → return empty state
* If `workerId` changes → refetch automatically

---

## 4. Phase 2 – Refactor All Data Hooks (Critical)

### Required Changes

❌ Old (Invalid)

```ts
useDailyCollections()
```

✅ New (Required)

```ts
useDailyCollections(workerId)
```

Inside the hook:

```ts
useEffect(() => {
  if (!workerId) return;

  fetchData(workerId);
}, [workerId]);
```

### Tables Affected

* daily_collections
* weekly_expenses
* settlements
* settlement_allocations
* expense_payments
* payment_events

---

## 5. Phase 3 – Enforce Worker-Scoped Queries

### Rule

Every Supabase query MUST include:

```ts
.eq('worker_id', workerId)
```

### Example

```ts
supabase
  .from('weekly_expenses')
  .select('*')
  .eq('worker_id', workerId);
```

No exceptions.

---

## 6. Phase 4 – Worker Switch = Full Data Rehydration

### Required Behavior

When `activeWorkerId` changes:

1. Clear all worker-dependent state
2. Show skeleton UI
3. Re-run all data hooks
4. Recompute all derived values

### Explicit Anti-Pattern

❌ Reusing cached data across workers

---

## 7. Phase 5 – Centralized Worker-Aware Hooks

### Recommendation

Create worker-aware wrappers:

```ts
const { activeWorkerId } = useWorker();
const daily = useDailyCollections(activeWorkerId);
```

This makes worker dependency explicit and visible.

---

## 8. Phase 6 – Guardrails & Debugging Aids

### Mandatory Runtime Guards

```ts
if (!workerId) {
  console.warn('Data hook called without workerId');
  return [];
}
```

### Dev-Time Assertion

Log worker changes:

```ts
useEffect(() => {
  console.log('Active Worker Changed:', workerId);
}, [workerId]);
```

---

## 9. Phase 7 – UI Validation Checklist

When switching workers:

* Grid dates change correctly
* Weekly expenses change correctly
* Totals recompute correctly
* No stale data flashes

If ANY item fails → worker scoping is broken.

---

## 10. Definition of Done (Hard Requirements)

* Switching worker always changes visible data
* Refresh restores the same worker + data
* No data from Worker A appears in Worker B
* No hook runs without `activeWorkerId`
* Skeletons show during rehydration

---

## 11. Mental Model (For Long-Term Stability)

> **Worker switching is equivalent to switching databases.**

If something does not change when switching workers, it is a bug.
