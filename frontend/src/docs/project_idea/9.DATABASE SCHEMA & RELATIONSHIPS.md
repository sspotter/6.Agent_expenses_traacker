# 🗄️ DATABASE SCHEMA & RELATIONSHIPS

## Daily Income, Expense & Debt Tracking System

---

## 1. Purpose of This Document

This document describes:

* All database tables used in the system
* What each table represents (business meaning)
* How tables are connected
* Which tables are **immutable**, **append-only**, or **editable**
* How money flows through the system

This is a **conceptual + structural reference**, not a migration file.

---

## 2. High-Level Data Model Overview

```
User
│
├── DailyCollection
│     └── SettlementAllocation
│            └── Settlement
│
├── Expense
│     └── ExpensePayment
│
└── PaymentEvent (audit log)
```

Key idea:

* **Income flow** and **Expense flow** are separate
* Debt is derived, not stored
* History is never rewritten

---

## 3. Core Tables

---

## 3.1 DailyCollection

📌 *Immutable daily income record*

Represents **what happened on a specific calendar day**.

### Purpose

* Track expected vs collected income
* Identify full / partial / missed days
* Act as the **source of income debt**

### Fields (conceptual)

| Field            | Description                       |
| ---------------- | --------------------------------- |
| id               | Primary key                       |
| user_id          | Owner                             |
| date             | Unique per user                   |
| expected_amount  | Expected income                   |
| collected_amount | Income received that day only     |
| status           | FULL / PARTIAL / MISSED (derived) |
| notes            | Optional                          |
| created_at       | Timestamp                         |

### Rules

* ❌ Never edited once the month ends
* ❌ Never updated by settlements
* ✅ Read-only for past months

---

## 3.2 Settlement

📌 *Late income received in the future*

Represents income **received after the original day**.

### Purpose

* Capture delayed payments
* Reduce income debt
* Count as income in the settlement month

### Fields

| Field           | Description     |
| --------------- | --------------- |
| id              | Primary key     |
| user_id         | Owner           |
| settlement_date | Always “today”  |
| amount          | Amount received |
| source_month    | Reference only  |
| notes           | Optional        |
| created_at      | Timestamp       |

### Rules

* ❌ Never edits DailyCollection
* ❌ Never used to pay expenses
* ✅ Always creates allocations

---

## 3.3 SettlementAllocation

📌 *Links settlements to income debt*

### Purpose

* Explain **where settlement money went**
* Prevent over-settling
* Enable audit trail

### Fields

| Field               | Description                |
| ------------------- | -------------------------- |
| id                  | Primary key                |
| settlement_id       | FK → Settlement            |
| daily_collection_id | FK → DailyCollection       |
| amount_applied      | Amount applied to that day |

### Rules

* ✅ Many-to-many bridge
* ✅ Append-only
* ❌ Never exceeds remaining balance

---

## 4. Expense Tables

---

## 4.1 Expense

📌 *Cost incurred (not necessarily paid)*

### Purpose

* Track weekly or one-off expenses
* Create **expense debt**

### Fields

| Field      | Description   |
| ---------- | ------------- |
| id         | Primary key   |
| user_id    | Owner         |
| date       | Expense date  |
| week_start | Week grouping |
| week_end   | Week grouping |
| amount     | Total cost    |
| category   | Optional      |
| notes      | Optional      |
| created_at | Timestamp     |

### Rules

* ❌ Does NOT affect income
* ❌ Does NOT imply payment
* ✅ Creates expense debt until paid

---

## 4.2 ExpensePayment

📌 *Money paid to clear expenses*

### Purpose

* Track partial or full payments of expenses
* Reduce expense debt

### Fields

| Field        | Description  |
| ------------ | ------------ |
| id           | Primary key  |
| expense_id   | FK → Expense |
| payment_date | Date paid    |
| amount       | Amount paid  |
| notes        | Optional     |
| created_at   | Timestamp    |

### Rules

* ❌ Not a settlement
* ❌ Does not affect collected income
* ✅ Reduces expense debt only

---

## 5. Audit & Transparency

---

## 5.1 PaymentEvent

📌 *Append-only audit log for explanations*

### Purpose

* Explain **how** something was paid
* Generate human-readable payment history

### Fields

| Field        | Description                 |
| ------------ | --------------------------- |
| id           | Primary key                 |
| entity_type  | DAILY_COLLECTION / EXPENSE  |
| entity_id    | FK to entity                |
| event_type   | COLLECTED / SETTLED / PAID  |
| amount       | Amount involved             |
| event_date   | Date                        |
| reference_id | FK to settlement or payment |
| created_at   | Timestamp                   |

### Rules

* ✅ Append-only
* ❌ Never edited
* ❌ Never deleted

---

## 6. Derived Concepts (NOT Tables)

These **must never be stored**.

---

### 6.1 Income Debt

```
SUM(expected_amount - collected_amount)
- SUM(settlement_allocations.amount_applied)
```

Derived from:

* DailyCollection
* SettlementAllocation

---

### 6.2 Expense Debt

```
SUM(expense.amount)
- SUM(expense_payments.amount)
```

Derived from:

* Expense
* ExpensePayment

---

### 6.3 Net Position

```
(Collected income + Settlements)
- Expenses paid
```

---

## 7. Table Mutability Summary

| Table                | Editable?                   | Notes              |
| -------------------- | --------------------------- | ------------------ |
| DailyCollection      | ⚠️ Current month only       | Locked after month |
| Settlement           | ✅ Append-only               | Never edited       |
| SettlementAllocation | ✅ Append-only               | Never edited       |
| Expense              | ⚠️ Editable (current month) | Safe updates       |
| ExpensePayment       | ✅ Append-only               | Never edited       |
| PaymentEvent         | ❌ Never                     | Audit log          |

---

## 8. Referential Integrity (Important)

* SettlementAllocation → requires valid Settlement & DailyCollection
* ExpensePayment → requires valid Expense
* PaymentEvent → references immutable records

Foreign keys should **restrict deletes**.

---

## 9. Mental Model Summary

```
Income
 ├── DailyCollection (on time)
 └── Settlement (late)

Expenses
 ├── Expense (incurred)
 └── ExpensePayment (paid)

Debt
 ├── Income debt (partials / missed)
 └── Expense debt (unpaid)

Audit
 └── PaymentEvent (explains everything)
```
