# 🔁 Phase: Settlement Attribution & Completion Logic

## Eliminating Double Counting & Misleading Daily States

---

## 1. Background & Observed Issue

### Current Behavior

* Daily cards show:

  * `$1 / 100`
  * Status: **PARTIAL** (yellow)
* Monthly summary shows:

  * Expected: `$200`
  * Collected: `$2`
  * Outstanding: `$198`

When a **settlement of $198** is recorded:

* Monthly summary updates correctly:

  * Collected increases
  * Outstanding becomes `$0`
  * Completion updates
* ❌ **Daily cards remain unchanged** (`$1 / 100`, PARTIAL)

If the user then **manually updates the daily cards**:

* Collected becomes `$398`
* ❌ Settlement is counted twice

---

## 2. Root Cause (Design-Level)

The system currently violates this rule:

> **A single unit of money must belong to exactly one source of truth.**

### What’s going wrong

* Daily cards are showing **only same-day collections**
* Monthly summary includes **settlements**
* UI suggests settlements should “complete” past days
* Manual edits attempt to “fix” visuals → double counting

This is not a calculation bug — it is a **representation & attribution problem**.

---

## 3. Non-Negotiable Accounting Rule (Locked)

### 🔒 Rule 1 — Settlements NEVER modify DailyCollection

* `DailyCollection.collected_amount`
  represents **money received on that exact day**
* Settlements:

  * Do NOT edit daily rows
  * Do NOT change daily status
  * Do NOT “fill in” yellow cards

This preserves:

* Immutability
* Audit correctness
* Time-accurate history

---

## 4. New Concept: Effective Completion vs Daily Status

We must **separate two ideas** that were previously conflated.

---

### 4.1 Daily Status (Immutable, Historical)

Derived ONLY from:

```
DailyCollection.collected_amount
```

| Condition                | Status  |
| ------------------------ | ------- |
| collected == expected    | FULL    |
| 0 < collected < expected | PARTIAL |
| collected == 0           | MISSED  |

🔒 **This status NEVER changes after the day/month closes**

---

### 4.2 Income Debt Resolution (Cross-Day, Cross-Month)

Settlements reduce **income debt**, not daily records.

Debt is derived as:

```
SUM(expected - collected)
- SUM(settlement_allocations)
```

Settlements answer:

> “Has the missing money been paid *eventually*?”

Not:

> “Was this day fully collected on that day?”

---

## 5. UI Rule Changes (Critical)

### 5.1 Daily Cards MUST NOT Auto-Update After Settlement

Yellow cards **remain yellow** forever.

However, they gain **context**.

---

### 5.2 New Visual Indicator: “Debt Resolved”

Daily cards may display:

* 🟡 PARTIAL
* 🟡 PARTIAL — **Debt Settled Later**

Example UI copy:

```
$1 / 100
PARTIAL
Settled later via Apr 10 settlement
```

This information comes from:

* `SettlementAllocation`
* `PaymentEvent`

---

## 6. Manual Editing Guardrails (Hard Rules)

### 🚫 Rule 2 — Manual edits must NEVER include settlement money

If a day has:

* Collected: `$1`
* Settlement allocated later: `$99`

The UI must:

* Disable increasing `collected_amount` beyond original day
* Show warning if attempted:

> “This day was already settled later.
> Editing will cause double counting.”

---

## 7. Completion Percentage Redefinition (IMPORTANT)

### Problem

Currently:

```
completion = collected / expected
```

This breaks when:

* Collected > Expected
* Settlements exist
* Over-collection happens

---

### 7.1 New Completion Definition

Completion is **progress**, not a cap.

```
completion = total_collected / total_expected
```

Where:

```
total_collected =
SUM(DailyCollection.collected_amount)
+ SUM(Settlement.amount in period)
```

---

### 7.2 Completion Rules

| Scenario                    | Completion |
| --------------------------- | ---------- |
| Expected 200, Collected 200 | 100%       |
| Expected 200, Collected 250 | 125%       |
| Expected 200, Collected 398 | 199%       |

✅ Completion **CAN exceed 100%**

---

### 7.3 UI Display Rules

* ≤100% → normal progress bar
* > 100% → overflow style

Example:

```
Completion: 199%
Over-collected by $198
```

This aligns with real accounting reality.

> **Daily records explain *when* money was received.
> Settlements explain *when debt was resolved*.
> Completion explains *how far beyond expectations you went*.**

If these three ever mix, the system is wrong.
