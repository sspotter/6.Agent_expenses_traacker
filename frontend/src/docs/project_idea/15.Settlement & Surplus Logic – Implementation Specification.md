# Settlement & Surplus Logic – Implementation Specification

This document defines the **authoritative rules and data flow** for handling daily collections, surplus, and settlements. It is written to be directly implementable on top of the existing Supabase schema and UI.

---

## 1. Core Principles (Locked)

These rules are **non-negotiable** and must be enforced in both backend logic and UI.

1. **No auto-settlement**

   * Surplus money must NEVER automatically cover missed or partial days.
   * All settlements are explicit admin actions.

2. **Surplus scope**

   * Surplus is tracked **per worker, per month**, and starts as **unallocated**.

3. **Daily records are immutable facts**

   * Original daily collection records are never overwritten by settlements.
   * Settlements only augment totals.

4. **Non-working days are not settleable**

   * Days with `expected_amount = 0` are excluded from settlement logic.
   * They may generate surplus if collected > 0.

5. **Settlements are granular**

   * A settlement may span multiple days.
   * Each target day gets its own settlement record.

---

## 2. Day Classification (Must Exist in Data)

Every calendar day must fall into exactly one category.

### A. Working Day

Condition:

* `expected_amount > 0`

Allowed states:

* FULL
* PARTIAL
* MISSED

Financial behavior:

* Contributes to monthly expected
* Can have outstanding balance
* Can receive settlements

---

### B. Non-Working Day

Condition:

* `expected_amount = 0`

Financial behavior:

* Excluded from monthly expected
* Cannot be settled
* Cannot be MISSED or PARTIAL
* If `collected_amount > 0` → creates surplus

UI:

* Greyed out
* No settlement actions available

---

## 3. Daily Collection Table (Existing)

`daily_collections`

Required fields:

* `worker_id`
* `date`
* `expected_amount`
* `collected_amount`
* `status` (FULL | PARTIAL | MISSED)
* `is_expected_override` (boolean)

Rules:

* `expected_amount` is date-specific and immutable after day closes
* Overrides apply only to that date

---

## 4. Surplus Logic (New Concept)

### Definition

A surplus exists when:

```
collected_amount > expected_amount
```

Surplus amount:

```
surplus = collected_amount - expected_amount
```

### Storage Strategy

Surplus is tracked **per worker, per month**, not per day.

Recommended table:

`monthly_surplus`

* `id`
* `worker_id`
* `month`
* `year`
* `amount_total`
* `amount_unallocated`
* `created_at`

Rules:

* Surplus does NOT modify any daily record
* Surplus does NOT affect outstanding balances

---

## 5. Settlement Logic (New Table)

### Purpose

Settlements represent **explicit allocation of money** to missed or partial working days.

### Table: `settlements`

Fields:

* `id`
* `worker_id`
* `target_date`
* `amount`
* `source` (SURPLUS | CASH | OTHER)
* `source_reference_id` (nullable)
* `created_at`

Rules:

* One settlement record per target day
* Settlement amount cannot exceed that day’s outstanding
* Non-working days cannot be targets

---

## 6. Outstanding Calculation (Per Day)

For a working day:

```
outstanding = expected_amount
             - collected_amount
             - sum(settlements for that day)
```

Constraints:

* `outstanding >= 0`
* If outstanding = 0 → day is resolved

---

## 7. Monthly Expected & Completion

### Monthly Expected

```
sum(expected_amount)
FOR days WHERE expected_amount > 0
```

### Monthly Collected

```
sum(collected_amount) + sum(settlements.amount)
```

### Completion %

```
(monthly_collected / monthly_expected) * 100
```

Important:

* Surplus is NOT included unless settled
* Non-working days are excluded entirely

---

## 8. UI Enforcement Rules

### Daily Card

* Show surplus badge if collected > expected
* Red days remain red until settled

### Settlement UI

* Admin-only
* Select target days
* Select source (surplus or cash)
* Preview effect before confirming

### Surplus Visibility

* Show unallocated surplus per month
* Show settlement history
