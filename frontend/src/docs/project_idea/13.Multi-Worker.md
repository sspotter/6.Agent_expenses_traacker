Implementation Phase: Multi-Worker (User) Support
Goal

Support multiple workers, each with independent income, expenses, and settlements, with:

Ultra-smooth switching

No hard auth complexity

Zero impact on financial integrity

Clean UX with skeleton loading

Phase 1: Concept & Rules (Foundation)
Core Decisions

Workers are data partitions, not real accounts.

Only one active worker context at a time.

Switching workers never mutates data — it only changes the active scope.

Golden Rules

Every financial record belongs to exactly one worker

No cross-worker calculations

Switching workers resets derived state only, not raw data

Default worker has no password

Phase 2: Data Model Extension
New Entity: workers
Worker {
  id: uuid
  name: string
  avatar_url?: string
  password_hash?: string | null
  created_at
}

Data Model Changes (Mandatory)

Add worker_id to all financial tables:

daily_collections

weekly_expenses

settlements

settlement_allocations

expense_payments

Constraints

All queries must be filtered by worker_id

No record may exist without a worker

Phase 3: Active Worker Context (Critical)
Global State

Create a Worker Context Provider:

ActiveWorkerContext {
  workerId
  workerName
  avatarUrl
}

Responsibilities

Load worker from:

LocalStorage (fast)

Fallback to default worker

Expose:

switchWorker(workerId)

setWorkerDetails()

Why This Matters

This ensures:

Smooth switching

No page reload

All hooks react instantly

Phase 4: Smooth Worker Switching UX
UI Entry Point

Top-Right Round Avatar Icon

Shows worker avatar or initials

Always visible

Dropdown Menu (on click)

Edit Worker Details

Switch Users

Phase 5: Switch Users Modal
Modal Contents

List of existing workers

Each item:

Avatar

Name

Tap to switch (instant)

Add User Section

At the bottom:

Field	Rule
Name	Required
Password	Optional (default: none)
Behavior

Create worker

Auto-switch to new worker

Persist to local storage

Phase 6: Password Logic (Simple & Local)
Rules

Password is optional

Stored as hash

Used only when switching workers

No server auth sessions

Flow

If worker has password → prompt once

Cache unlock in memory for session

No re-prompt unless page reloads

Phase 7: Data Loading & Skeletons
On Worker Switch

Clear derived UI state

Show page skeleton

Fetch worker-scoped data

Hydrate UI

Fade-in content

Skeleton Coverage

Calendar grid

Monthly summary

Weekly expenses

Debt explorer

⚠️ No spinners — skeletons only

Phase 8: Hook & Query Refactor
Mandatory Changes

All data hooks must accept workerId:

useDailyCollections(workerId)
useWeeklyExpenses(workerId)
useSettlements(workerId)

Derived Calculations

No change needed

Calculations remain valid per worker

Phase 9: Performance & UX Polish
Optimizations

Cache last worker’s data

Preload avatars

Optimistic worker switching

UX Guarantees

Switching feels < 300ms

No full reload

No flashing UI

Phase 10: Edge Cases & Safeguards
Must Handle

Deleting a worker (soft delete only)

Worker with zero data

Switching mid-settlement flow → cancel action

Invalid password attempt (non-blocking)

Final Mental Model (Very Important)

Workers are lenses, not users.
The app never logs in — it simply changes perspective.

This keeps:

Your accounting rules pure

Your UX lightning-fast

Your future scalability wide open
